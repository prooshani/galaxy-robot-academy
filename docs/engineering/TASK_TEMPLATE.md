# Engineering Task Template

Every Engineering Work Order should use the following structure.

## Objective

State the single outcome the task must achieve and how success will be recognized.

## Background

Explain the product or technical reason for the work and any relevant prior decisions.

## Repository Context

Identify the relevant applications, packages, modules, data flows, documentation, and existing patterns that the engineer must inspect.

## Requirements

List the behaviors, interfaces, content, and quality expectations that must be implemented.

## Constraints

Define fixed boundaries such as approved technologies, compatibility requirements, architecture decisions, security rules, and prohibited changes.

## Expected Files

List files likely to be created, modified, or deleted. This guides scope but does not replace repository investigation.

## Acceptance Criteria

Provide objective, testable statements that must all be true for the task to be complete.

## Validation

Specify the commands, tests, manual checks, and evidence required to demonstrate correctness.

## Deliverables

List every artifact expected from the task, including code, tests, documentation, migrations, assets, and the Task Report.

## Out of Scope

State adjacent work that must not be included, preventing assumptions and scope drift.
